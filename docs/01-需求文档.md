# 宠物管理与服务平台 - 需求文档

## 1. 项目概述

### 1.1 项目背景
随着城市化进程的加速与居民生活水平的提高，宠物在家庭中的角色日益重要，从传统的看家护院转变为情感伴侣。然而，当前宠物领养、寻宠、社区交流等服务仍依赖线下管理，存在信息分散、流程不规范、效率低下等问题。

### 1.2 项目目标
构建一个功能完整、交互友好的宠物管理与服务平台，实现宠物档案管理、领养流程线上化、走失宠物互助发布以及社区互动交流的一站式服务。

### 1.3 项目价值
- 提升宠物领养流程的透明度和效率
- 促进流浪动物的领养和救助
- 增强社区宠物互助能力
- 推动宠物服务行业的数字化发展

## 2. 用户角色分析

### 2.1 普通用户（宠物爱好者/领养者/失主）

**角色描述**：平台的主要使用群体，包括寻求领养宠物的主人、走失宠物的失主、以及参与社区互动的宠物爱好者。

**核心需求**：
- 浏览待领养宠物信息
- 在线提交领养申请
- 发布走失宠物信息
- 发布和浏览社区动态
- 参与评论和点赞互动

**权限范围**：
- 查看所有公开信息
- 管理个人发布的内容
- 查看个人的申请记录
- 修改个人资料

### 2.2 管理员

**角色描述**：平台的管理和运营人员，负责内容审核、流程管理和平台维护。

**核心需求**：
- 管理宠物档案信息
- 审核领养申请
- 维护走失宠物信息
- 管理社区内容（删除违规内容）
- 查看平台运营数据

**权限范围**：
- 所有普通用户权限
- 宠物信息的增删改查
- 领养申请的审核和状态更新
- 违规内容的删除
- 用户管理权限

## 3. 功能需求详细说明

### 3.1 用户与权限管理模块

#### 3.1.1 用户注册
**功能描述**：新用户通过注册账号成为平台成员

**输入要素**：
- 用户名（唯一，3-20字符）
- 密码（6-20字符，需加密存储）
- 联系方式（手机号或邮箱）
- 昵称（2-20字符）

**业务规则**：
- 用户名不能重复
- 密码需进行哈希加密
- 注册成功后自动分配"普通用户"角色
- 默认头像为系统提供

**输出结果**：注册成功，自动登录

#### 3.1.2 用户登录
**功能描述**：已注册用户通过账号密码登录系统

**输入要素**：
- 用户名
- 密码

**业务规则**：
- 使用JWT进行身份验证
- 登录成功后生成token返回前端
- token有效期24小时

**输出结果**：登录成功，返回用户信息和token

#### 3.1.3 个人信息管理
**功能描述**：用户查看和修改个人资料

**可修改字段**：
- 昵称
- 联系方式
- 头像（支持图片上传）

**业务规则**：
- 用户名不能修改
- 头像上传需限制大小（≤5MB）
- 支持格式：jpg、jpeg、png

#### 3.1.4 权限控制
**功能描述**：系统根据用户角色控制功能访问

**权限矩阵**：

| 功能模块 | 普通用户 | 管理员 |
|---------|---------|--------|
| 浏览宠物信息 | ✓ | ✓ |
| 提交领养申请 | ✓ | ✓ |
| 发布走失信息 | ✓ | ✓ |
| 发布社区动态 | ✓ | ✓ |
| 评论点赞 | ✓ | ✓ |
| 管理宠物档案 | ✗ | ✓ |
| 审核领养申请 | ✗ | ✓ |
| 删除违规内容 | ✗ | ✓ |
| 用户管理 | ✗ | ✓ |

### 3.2 宠物档案管理模块

#### 3.2.1 宠物信息录入（管理员）
**功能描述**：管理员添加待领养宠物的详细信息

**必填字段**：
- 宠物名称（2-20字符）
- 品种（2-30字符）
- 性别（公/母）
- 年龄（数字，单位：岁）
- 健康状况（良好/一般/需治疗）
- 状态（待领养/申请中/已领养）
- 宠物照片（1-5张）

**选填字段**：
- 性格特点
- 疫苗接种情况
- 绝育情况
- 其他备注

**业务规则**：
- 照片大小限制：单张≤5MB
- 支持格式：jpg、jpeg、png
- 新录入宠物默认状态为"待领养"

#### 3.2.2 宠物信息编辑（管理员）
**功能描述**：管理员修改已录入的宠物信息

**可修改字段**：所有字段（除宠物ID外）

**业务规则**：
- 修改后需记录操作时间
- 已领养的宠物信息可查看但不可修改状态

#### 3.2.3 宠物信息删除（管理员）
**功能描述**：管理员删除无效或重复的宠物档案

**业务规则**：
- 仅可删除"待领养"状态的宠物
- "申请中"和"已领养"状态的宠物不可删除
- 删除前需二次确认

#### 3.2.4 宠物列表浏览（所有用户）
**功能描述**：用户浏览平台上的待领养宠物

**展示信息**：
- 宠物照片（主图）
- 宠物名称
- 品种
- 性别
- 年龄
- 状态标签

**支持筛选**：
- 按品种筛选
- 按性别筛选
- 按年龄范围筛选
- 按状态筛选

**支持排序**：
- 按录入时间倒序
- 按年龄正序/倒序

#### 3.2.5 宠物详情查看（所有用户）
**功能描述**：用户查看单个宠物的完整信息

**展示内容**：
- 所有基本信息
- 所有照片（可轮播查看）
- 宠物状态
- 如已领养，显示领养时间

### 3.3 领养申请管理模块

#### 3.3.1 提交领养申请（普通用户）
**功能描述**：用户对心仪的宠物提交领养申请

**必填字段**：
- 宠物ID
- 申请理由（10-500字符）
- 养宠经验（无/有，如有需描述）
- 联系方式（电话/微信）
- 居住地址

**业务规则**：
- 每个用户对同一宠物只能提交一次申请
- 仅可对"待领养"状态的宠物提交申请
- 提交后宠物状态自动变为"申请中"
- 同一用户同时最多有3个待审核的申请

**输出结果**：申请提交成功，等待管理员审核

#### 3.3.2 查看我的申请（普通用户）
**功能描述**：用户查看自己提交的所有领养申请

**展示信息**：
- 申请时间
- 宠物基本信息
- 申请内容
- 审核状态（待审核/已通过/已驳回）
- 审核意见（如已审核）

**筛选功能**：
- 按审核状态筛选
- 按申请时间排序

#### 3.3.3 申请列表查看（管理员）
**功能描述**：管理员查看所有领养申请

**展示信息**：
- 申请用户信息
- 宠物基本信息
- 申请内容
- 申请时间
- 审核状态

**筛选功能**：
- 按审核状态筛选
- 按申请时间排序
- 支持按用户名、宠物名称搜索

#### 3.3.4 申请审核（管理员）
**功能描述**：管理员对申请进行审核

**审核操作**：
- 通过：批准领养申请
- 驳回：拒绝领养申请

**通过操作的业务规则**：
- 将宠物状态更新为"已领养"
- 记录领养时间
- 记录领养人信息
- 驳回该宠物的其他所有申请
- 通知申请用户

**驳回操作的业务规则**：
- 将宠物状态恢复为"待领养"
- 需填写驳回原因（10-200字符）
- 通知申请用户
- 用户可重新提交申请

**输出结果**：审核完成，状态更新，发送通知

### 3.4 走失宠物互助模块

#### 3.4.1 发布走失信息（所有用户）
**功能描述**：用户发布走失宠物信息，寻求社区帮助

**必填字段**：
- 宠物名称
- 走失地点（省市区+详细地址）
- 走失时间（日期+时间）
- 宠物描述（外观特征、品种、年龄等，20-500字符）
- 联系方式（电话/微信）
- 宠物照片（至少1张，最多3张）

**可选字段**：
- 是否紧急（是/否，默认否）

**业务规则**：
- 照片大小限制：单张≤5MB
- 支持格式：jpg、jpeg、png
- 紧急信息在前台列表中置顶并标识
- 用户发布的信息可由本人编辑或删除

**输出结果**：发布成功，信息显示在走失列表

#### 3.4.2 浏览走失信息（所有用户）
**功能描述**：用户浏览所有走失宠物信息

**展示信息**：
- 宠物照片
- 宠物名称
- 走失地点
- 走失时间
- 紧急标识（如标记为紧急）
- 发布时间

**列表排序**：
- 紧急信息优先
- 按发布时间倒序

#### 3.4.3 走失详情查看（所有用户）
**功能描述**：查看走失宠物的完整信息

**展示内容**：
- 所有发布信息
- 所有照片
- 距离发布时间（如：3小时前）
- 联系方式（需登录后查看）

#### 3.4.4 我的走失信息（普通用户）
**功能描述**：用户查看自己发布的走失信息

**操作功能**：
- 编辑信息
- 删除信息
- 标记为已找到（删除并记录）

#### 3.4.5 走失信息管理（管理员）
**功能描述**：管理员维护走失信息列表

**管理功能**：
- 删除违规信息
- 删除过期信息（超过30天）
- 删除已找到的信息

## 4. 非功能性需求

### 4.1 性能需求
- 页面加载时间：首页≤2秒
- API响应时间：普通查询≤500ms，复杂查询≤1s
- 并发支持：至少支持100个并发用户

### 4.2 安全需求
- 密码加密存储（使用bcrypt）
- JWT token身份验证

- XSS攻击防护
- 文件上传类型和大小限制
- 接口访问频率限制（防止暴力攻击）

### 4.3 可用性需求
- 系统可用性：99%以上
- 界面友好，操作简单直观
- 关键操作提供确认提示
- 错误信息清晰易懂

- 代码结构清晰，模块化设计

### 4.5 可扩展性需求

- 数据库设计考虑未来扩展

## 5. 数据模型概述

### 5.1 核心实体关系

```
用户（sys_user）
  ├── 1:N → 宠物信息（pet_info）[管理员发布]
  ├── 1:N → 领养申请（adoption_application）[用户提交]
  ├── 1:N → 走失信息（lost_pet）[用户发布]
  ├── 1:N → 社区帖子（forum_post）[用户发布]
  ├── 1:N → 评论（forum_comment）[用户发表]
  └── 1:N → 点赞（forum_like）[用户操作]

宠物信息（pet_info）
  └── 1:N → 领养申请（adoption_application）

社区帖子（forum_post）
  ├── 1:N → 评论（forum_comment）
  └── 1:N → 点赞（forum_like）

评论（forum_comment）
  └── 1:N → 子评论（forum_comment，自关联）
```

### 5.2 主要数据表

| 表名 | 说明 | 主要字段 |
|-----|------|---------|
| sys_user | 用户表 | id, username, password, nickname, avatar, role, contact_info |
| pet_info | 宠物信息表 | id, name, breed, gender, age, health_status, status, photos |
| adoption_application | 领养申请表 | id, user_id, pet_id, reason, experience, contact, address, status |
| lost_pet | 走失宠物表 | id, user_id, name, location, lost_time, description, photos, is_urgent |
| forum_post | 社区帖子表 | id, user_id, title, content, images |
| forum_comment | 评论表 | id, post_id, user_id, content, parent_id |
| forum_like | 点赞表 | id, post_id, user_id |

## 6. 业务流程说明

### 6.1 领养流程
1. 用户浏览待领养宠物列表
2. 选择心仪宠物查看详情
3. 填写领养申请表单并提交
4. 宠物状态变为"申请中"
5. 管理员在后台查看申请列表
6. 管理员审核申请（通过/驳回）
7. 若通过：宠物状态变为"已领养"，通知用户
8. 若驳回：宠物状态恢复"待领养"，填写驳回原因，通知用户

### 6.2 寻宠流程
1. 用户宠物走失，登录平台
2. 发布走失信息（填写详细信息、上传照片）
3. 信息发布到走失列表（紧急信息置顶）
4. 其他用户浏览并提供线索
5. 联系发布者
6. 找到宠物后，发布者删除信息或标记已找到

### 6.3 社区互动流程
1. 用户发布养宠相关的动态
2. 动态显示在社区列表
3. 其他用户浏览、点赞、评论
4. 支持楼中楼回复（最多2层）
5. 管理员监督内容，删除违规信息
